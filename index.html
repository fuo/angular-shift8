Bootstrap Modules for use with Twitter Bootstrap...

Using Twitter Bootstrap? Well, as you know it has a specific HTML structure for forms and various components. I've created a set of directives that help make this an easier process which will speed up your development time.

Forms /modules/bootstrap/form.js

Let's start with some directives for creating forms quickly. They are <b-form>, <b-field>, and <b-form-progress>.
You'll typically want to nest the fields and progress bar within the form directive.


A note about select, radio, and checkbox options...
Angular.js has an attribute ng-options which is not the easiest thing to use. I'm using a convention and sticking to a specific format in my directive templates. This isn't to say that you can't ignore my options attribute and write your own ng-options to have that passed along and pull from some variable in your controller...But I would highly recommend using my format (seen above in the help area under some of the fields).

The format is pretty straight forward. I use an object (not an array) even for simple lists. Using an array (by Angular default) will assign numbers for the values, for example:
<option value="0">Option Label</option>
Which I think is just weird and not very useful. How many times do you want a number stored in your database that only references the order in which you listed values on the front-end?! So yea, key those values and use an object...But even still Angular gives you a few ways to go about that. I came up with an object format where the key is the option value and then the value is another object with an option key of which the value is used for the label/text that user sees in the select/radio/checkbox choice. Then optional group and selected or checked keys that have true/false for selected/checked and a string value for select groups.

Use this convention for checkboxes, select lists, and radio buttons and switch between the form element types without changing your object! This makes things a lot more sane in my opinion, but again you are free to do as you wish.

Templates

If you need to, you can change the directive templates and then switch which field is using which template on a case by case basis. Let's say you didn't like how I wired the HTML for the text input field...Or maybe you needed to break Twitter Bootstrap's structure. You could write your own template, calling it text.html and specify a new base path for the template. For example: <b-field type="text" ng-model="MyField" template-base-path="full-url.com/path/or/from/site/root/"></b-field>

The directive templates are dynamically loaded in via $http call in Angular, so it can be pretty much anywhere. The name of the template will be based off of the type attribute. By default, I've provided; text, password, select, radio, checkbox, file, and textarea. However, you can create your own custom input field types. Think date pickers, etc. You'll just need to create the templates and define the new base path.

Compatibility

I love Angular-UI, it's utils and Angular-Bootstrap. This form directive is fully compatible so you can use attributes such as ui-validate and ui-jq="setMask" with ui-options='{mask: "(999) 999-9999"}' for example.

In fact, I built this directive with Angular-UI in mind. So you can use validation and have the rule key name transform into the error message. For example: ui-validate="{'You must provide a valid e-mail address.': 'validate.email($value)'}"

In the example above, I have a validate object in my controller with an email property that has a function that runs a regex to check an e-mail address format. The function returns true/false. See the Angular-UI documention for more information on validation and such. However, the key here is used in my directive templates to display to the user. By default, Angular-UI calls the key validate if not specified, which I do not display because it doesn't make sense. So key your validation rules.
